Known Issues

Current limitations and production hardening guidance.

⚠️
OpenFactstore is under active development. The items below represent known limitations in the current release. Each entry includes recommended mitigations for production use.

Current Limitations

Security No built-in authentication

All API endpoints are unauthenticated. Any client with network access can read or write compliance data, including attestations, approvals, and policy definitions.

Mitigation: Place OpenFactstore behind a reverse proxy (nginx, Caddy, or an API gateway) that enforces authentication — such as HTTP Basic Auth, OAuth 2.0 bearer tokens, or mTLS — before any request reaches the application. Do not expose the API port directly to untrusted networks.

Limitation In-memory H2 database

The default configuration uses an H2 in-memory database. All compliance data — flows, trails, attestations, and audit records — is lost when the application restarts.

Mitigation: Configure an external PostgreSQL database by setting the Spring datasource properties in application.properties. The JPA schema is compatible with PostgreSQL without code changes. A file-mode H2 database (jdbc:h2:file:./factstore) is a lower-effort interim option for persistence without a separate database server.

Security SCM token storage

Source Control Management (SCM) tokens used for pull-request attestation are stored as Base64-encoded strings. This is obfuscation, not encryption — the values are recoverable from the database without a key.

Mitigation: Use a secrets manager such as HashiCorp Vault, AWS Secrets Manager, or Azure Key Vault to store tokens externally. Inject them at runtime via environment variables rather than persisting them in the database.

Limitation OPA embedded mode is a simplified stub

The default embedded OPA policy evaluator supports a simplified subset of policy logic. Full Rego language features — including imports, virtual documents, and partial rules — are not available in embedded mode.

Mitigation: Set OPA_MODE=external and configure OPA_URL to point to a full OPA server instance. Full Rego support is available in external mode.

Limitation No RBAC

Role-based access control is not yet implemented. All operations — including creating and approving compliance flows, managing organisations, and evaluating deployment gates — are globally accessible to any authenticated caller.

Mitigation: Use network-level segmentation to restrict which services and users can reach sensitive endpoints. RBAC is on the roadmap for a future release.

Limitation Incomplete frontend coverage

Several backend features do not yet have dedicated frontend views in the Vue application. Affected areas include the regulatory compliance framework (SOX, PCI-DSS, GDPR controls), OPA policy management, and environment drift detection.

Mitigation: Use the REST API directly or the Swagger UI at /swagger-ui.html to interact with these features. Frontend coverage will be expanded in upcoming releases.

Planned Enhancements

The following capabilities are on the roadmap for upcoming releases:

Planned CLI binary (issue #33)

A standalone CLI tool for interacting with the OpenFactstore API from shell scripts and CI/CD pipelines, without requiring curl + jq.

Planned Full authentication + RBAC

JWT-based authentication with role-based access control, supporting at minimum admin, auditor, and pipeline-bot roles.

Planned PostgreSQL support

First-class PostgreSQL configuration profile with Flyway migration scripts and a Docker Compose profile for a persistent development environment.

Planned Kubernetes operator

A Kubernetes operator for managing OpenFactstore deployments declaratively, with custom resources for Flows and deployment policies.
